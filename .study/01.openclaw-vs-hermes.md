# openclaw vs hermes

## 1. 프롬프트 인젝션

- **openclaw**: "수상한 입력을 걸러내겠다" 는 식의 방어를 agent 안에 둠. 근데 LLM 입력이라는 게 결국 자연어라 규칙으로 다 못 거름.
- **hermes**: 인젝션은 일어난다고 가정. 대신 LLM 이 시키는 명령을 **어디서 실행하느냐** 를 통제 — 호스트가 아니라 컨테이너 / 샌드박스 안에서 돌게 함.

## 2. agent 하나 털리면 모든 권한 털림

- **openclaw**: agent 가 가진 모든 자격증명(API key, 토큰)이 한 덩어리. tool 이든 skill 이든 다 같이 봄.
- **hermes**: 세 가지 방어를 둠.
    1. **자식 프로세스로의 자동 전파 차단.** agent 가 shell / MCP / 코드 실행 자식 프로세스를 띄울 때, 자기 환경변수에 들어있던 LLM provider 키 / 메신저 토큰 / 클라우드 자격증명 등을 **자동으로 빼고** 넘김. 즉 LLM 이 인젝션으로 `env` 나 `echo $OPENAI_API_KEY` 같은 걸 시켜도 자식 프로세스 환경에 그 값이 없음.
    2. **skill 이 함부로 우회 못하게 막음.** skill 이 "나는 이 환경변수 필요해" 라고 선언할 수 있긴 한데, hermes 가 관리하는 핵심 자격증명 목록은 skill 선언으로도 통과 못 시킴. (실제로 과거에 악성 skill 이 `OPENAI_API_KEY` 를 passthrough 로 등록해서 빼간 CVE 가 있었고 그 이후로 막힘.)
    3. **외부 입구마다 화이트리스트 필수.** Telegram / Discord / Slack / API server 같은 외부 채널은 "허용된 발신자 리스트" 가 설정되기 전까진 요청을 거부함. 메신저 봇 토큰만 알아낸다고 agent 한테 명령 못 날림.
- **단, 한계:** agent **메인 프로세스 자체** 안에서는 여전히 평탄함. skill / plugin / hook 이 in-process 로 로드되면 agent 가 가진 자격증명을 다 읽을 수 있음. 그래서 hermes 도 "이 부분이 걱정되면 agent 통째로 컨테이너 / 샌드박스에 넣어라" 라고 말함 (꼭지 1 의 OS 격리와 같은 얘기).

## 3. 보안 설정 복잡도

- **openclaw**: 보안 옵션 여기저기 흩어짐. 뭘 켜야 어디까지 안전해지는지 사용자가 알아서 추론.
- **hermes**: 기본값은 똑같이 느슨함. 다만 "안전 모드" 가 두 가지로 정리돼 있음 — 명령 실행만 격리할지, agent 통째로 격리할지. 느슨하게 쓰는 건 "내가 알고 받아들이는 트레이드오프" 로 분류.

## 4. 써드파티 코드 신뢰 (skill / 플러그인 / MCP)

- **openclaw**: 마켓플레이스에서 까는 게 사실상 임의 코드 실행인데 그 점을 명확히 안 알림.
- **hermes**: "이거 깐다는 건 너가 검수한다는 뜻이다. 악성이면 그건 너 책임이다" 를 문서에 박음. 실제 위험은 똑같음, 책임 소재만 분리.

## 5. 외부 포트

- **openclaw**: 메신저 / API 어댑터마다 인증 정책 제각각.
- **hermes**: 모든 외부 노출 채널에 동일 규칙 — 화이트리스트 없으면 동작 거부, session ID 안다고 권한 생기지 않음.

---

## 종합

hermes 가 풀권한 agent 의 근본 문제를 **구조적으로 푼 건 없다**. 다음 두 가지를 했을 뿐:

1. agent 안에서 막는 것(필터, 패턴 검사)과 OS 가 막는 것(컨테이너, 샌드박스)을 분리해서, **agent 안의 방어는 사고 방지일 뿐 보안 경계가 아니다** 라고 명시.
2. 진짜 경계가 필요할 때 컨테이너 / 외부 샌드박스를 공식적으로 붙일 수 있게 해둠.

위협모델은 그대로 유효함.
